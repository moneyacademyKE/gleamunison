# ADR 0018: Process Dictionary for Dynamic Effect Handler Stacks

## Context
When compiling Unison's dynamic algebraic effects (e.g. `Do` and `Handle` terms) to Erlang BEAM bytecodes, the compiled code must support lexically scoped, nested effect handlers. When an operation is triggered via `do_op`, the runtime must locate the innermost handler that registers a callback for the given ability.

## Decision
We implement a thread-local effect handler stack in Erlang using the process dictionary (under the key `{gleamunison_handlers}`).
- **`handle_comp/2`** takes a handler value and a computation thunk. It pushes the handler onto the stack, executes the thunk, and restores the previous stack state in a `try ... after` block to guarantee safety even if the thunk throws an exception.
- **`do_op/4`** walks down the handler list stack from head to tail (innermost to outermost). The first handler matching the ability key (either atom or binary) and operation index handles the request.
- We support multiple handler formats (tuples, maps, and 3-arity functions) to ensure maximum compatibility with any compiled representations generated by the compiler.

## Consequences
- **Nesting**: Lexically nested handlers for the same or different abilities resolve correctly, with inner handlers shadowing outer ones.
- **Robustness**: The `try ... after` block ensures that if a computation crashes, the handler stack is restored immediately, preventing memory leaks or cross-request contamination in process pools.
- **No Message Overhead**: Using the process dictionary avoids spawning intermediate actor processes or sending IPC messages, maximizing performance.
