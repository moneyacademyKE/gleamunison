# Rich Hickey Gap Analysis: Nibble vs. Atto in Gleam

This analysis compares **Nibble** and **Atto**, the two most prominent parser combinator libraries in the Gleam ecosystem, to evaluate their suitability for compiler/runtime surface syntax parsing.

---

## 1. Architectural Strategy Comparison

| Capability | Nibble | Atto |
|---|---|---|
| **Design Philosophy** | **Elm-style Two-Phase Parser** (Lexer → Tokens → AST) | **Extensible Monadic Streams** (Parser directly on Input Stream) |
| **Stream Input Type** | Requires Token list generated by Lexer | Flexible: String character stream, binary bits, or custom tokens |
| **Backtracking** | Native backtracking controls (`nibble.backtrackable`) | Built-in automatic backtracking |
| **Pratt Parsing** | **Built-in** (Excellent for operator precedence) | Needs manual implementation or helper functions |
| **Error Quality** | Good error structures (based on token locations) | **Excellent** (Built-in pretty-printing context formatting) |
| **Execution Performance**| **High** (Lexer reduces string allocations before parsing) | Medium (Stream monad allocation overhead) |
| **Grammar Context** | Context-free | **Context-sensitive** (supports stateful/dependent grammars) |

---

## 2. In-Depth Feature Analysis

### Nibble: Two-Phase Lex/Parse
* **How it works**: You define a token type, a lexical tokenizer (e.g. mapping `"let"` to `LetToken`), and then compose parser combinators that match token sequences.
* **Benefits**:
  - Highly performant. The lexing pass converts raw strings to native Gleam constructors in a single pass. The parsing pass then operates on a lightweight `List(Token)` stream, reducing string slicing allocations.
  - Built-in Pratt parsing handles operator precedence (such as arithmetic expressions with `*` vs `+`) easily without complex left-recursion issues.
* **Trade-offs**:
  - Rigid workflow: You are forced to define a lexer token representation even for simple parse tasks.

### Atto: Stateful Monadic Streams
* **How it works**: Atto defines a generalized stream reader. You write parsers that match characters, substrings, or binary bits directly.
* **Benefits**:
  - High flexibility: You can parse context-sensitive syntax, binary files, or raw text streams without writing a middle token representation.
  - Exceptional diagnostics: Error objects capture state contexts automatically, offering pretty-printed diagnostics.
* **Trade-offs**:
  - Lower throughput: Composing monadic functions over character streams leads to high closure allocation overhead, making it slower on large files.

---

## 3. Complexity vs. Utility Matrix

| Library | Complexity (1-5) | Utility (1-5) | Recommendation |
|---|---|---|---|
| **Nibble** | 2 | 5 | **Best for standard programming languages**: Separating lexing from parsing matches typical compiler compiler pipelines (Elm style). |
| **Atto** | 3 | 4 | **Best for custom formats & binary protocols**: Ideal when parsing binary packets or highly dynamic, context-sensitive text templates. |

---

## 4. Actionable Recommendation

* If you are implementing a standard programming language with **infix operator precedence** and want optimal compiler performance, use **Nibble**. Its split lexer/parser design results in faster compilation passes and cleaner AST code.
* If you are writing a compiler/interpreter that needs **dynamic context-sensitive grammars** (where a token's meaning depends on previously parsed variables) or **binary protocols**, use **Atto**. 

For Gleamunison, our custom S-expression recursive-descent parser remains preferred because it introduces **zero package dependencies** and fits under the strict **150 LOC limit**.
